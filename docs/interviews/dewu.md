### 1. 网络不同层级的分工以及协议如何合作

OSI 七层 / TCP/IP 四层模型，每层各司其职：

| 层级 | 代表协议 | 职责 |
|------|---------|------|
| 应用层 | HTTP/HTTPS、DNS | 用户数据格式化 |
| 传输层 | TCP/UDP | 端到端可靠传输、流量控制 |
| 网络层 | IP、ICMP | 路由寻址 |
| 数据链路层 | Ethernet、ARP | 相邻节点传输 |
| 物理层 | 光纤、网线 | 比特流传输 |

**追问：举实际场景说明各层如何配合解决网络拥塞**

- **传输层（TCP）**：拥塞控制算法（慢启动、拥塞避免、快重传、快恢复），检测到丢包后降低发送窗口
- **网络层（IP/路由器）**：RED（随机早期检测）主动丢包，提前通知发送方降速
- **应用层（HTTP/2）**：多路复用，避免队头阻塞；HTTP/3（QUIC）在 UDP 上自己实现拥塞控制
- **数据链路层**：流量整形（Token Bucket），限制突发流量

### 2. 数据库锁机制

**核心作用：** 解决并发操作下的数据一致性问题（脏读、不可重复读、幻读、丢失更新）

**常见类别：**

| 锁类型 | 说明 | 适用场景 |
|--------|------|---------|
| 共享锁（S锁/读锁） | 多个事务可同时读，不能写 | 读多写少，数据查询 |
| 排他锁（X锁/写锁） | 只有一个事务可读写，其他全阻塞 | 写操作，需要强一致性 |
| 意向锁 | 表级锁，标记行锁意图 | InnoDB 内部使用 |
| 行锁 | 锁定单行 | 高并发，精细控制 |
| 表锁 | 锁定整张表 | 批量操作，MyISAM |
| 间隙锁（Gap Lock） | 锁定索引间隙，防止幻读 | RR 隔离级别 |
| 乐观锁 | 不加锁，提交时校验版本号 | 读多写少，冲突概率低 |

**追问1：排他锁在并发中的作用 + 电商库存扣减如何选锁避免超卖**

```sql
-- 方案1：悲观锁（SELECT FOR UPDATE）
BEGIN;
SELECT stock FROM goods WHERE id=1 FOR UPDATE;
UPDATE goods SET stock=stock-1 WHERE id=1 AND stock>0;
COMMIT;
-- 优点：强一致，不会超卖
-- 缺点：高并发下锁竞争严重，吞吐量低

-- 方案2：乐观锁（版本号/CAS）
UPDATE goods SET stock=stock-1, version=version+1
WHERE id=1 AND stock>0 AND version=#{version};

-- 方案3：Redis 原子操作（推荐高并发场景）
-- DECR stock_key（原子操作，天然防超卖）
```

**追问2：排他锁的性能问题**

- **锁等待**：其他事务全部阻塞，响应时间增加
- **死锁**：多个事务互相等待对方释放锁
- **锁升级**：行锁升级为表锁
- **吞吐量下降**：高并发下串行化执行，QPS 骤降

### 3. Python 深浅拷贝

| | 浅拷贝 | 深拷贝 |
|--|--------|--------|
| 方式 | `copy.copy()` / `list[:]` | `copy.deepcopy()` |
| 顶层对象 | 新建 | 新建 |
| 嵌套对象 | 共享引用（同一内存地址） | 完全独立（递归复制） |

```python
import copy

original = [[1, 2], [3, 4]]
shallow = copy.copy(original)
deep = copy.deepcopy(original)

original[0].append(99)
print(shallow)  # [[1, 2, 99], [3, 4]]  ← 受影响！
print(deep)     # [[1, 2], [3, 4]]      ← 不受影响
```

### 4. 冒烟测试

**定义：** 对新版本/新构建进行的快速基础功能验证，判断是否值得进行完整测试

**冒烟测试没通过怎么处理：**

1. **立即打回**：通知开发该版本不可测，附上失败截图/日志
2. **定位失败原因**：是新功能引入的 Bug，还是环境问题，还是用例本身问题
3. **记录缺陷**：提 Bug，标注严重级别（P0/P1）
4. **等待修复后重新冒烟**：不在不稳定版本上继续测试
5. **复盘**：推动开发先做 UT

---

### 5. 测试环境不稳定、问题复现困难

**处理思路：**

1. **记录现场**：截图、日志、请求/响应报文、时间戳
2. **环境排查**：服务状态、数据库连接、依赖服务、监控指标
3. **复现策略**：增加日志、固定数据、降低并发、MOCK 隔离
4. **沟通协作**：与运维确认环境、与开发确认变更
5. **长期方案**：推动搭建稳定环境、健康检查、数据工厂

### 6. JMeter 如何进行压测

**基本步骤：**

1. **创建测试计划** → 线程组：并发用户数、Ramp-Up 时间、循环次数
2. **添加 Sampler**：HTTP Request，配置接口地址、方法、参数
3. **添加配置元件**：HTTP Header Manager、CSV Data Set Config
4. **添加监听器**：聚合报告、响应时间图
5. **执行压测**：基准测试 → 逐步加压 → 观察 TPS、响应时间（P99）、错误率
6. **分析结果**：TPS 拐点 = 系统瓶颈，错误率 > 1% 需排查

### 7. Bug 优先级

| 级别 | 定义 | 示例 | 处理时限 |
|------|------|------|---------|
| P0（Blocker） | 系统崩溃、核心功能完全不可用 | 登录失败、支付崩溃 | 立即修复，阻塞上线 |
| P1（Critical） | 核心功能严重受损，无 workaround | 下单成功但库存不扣减 | 当天修复 |
| P2（Major） | 功能部分受损，有 workaround | 搜索结果排序错误 | 本迭代修复 |
| P3（Minor） | 体验问题、UI 瑕疵 | 按钮颜色偏差 | 下个迭代或酌情处理 |
